# `frontend + src` 下一次迭代灵感文档

## 1. 文档目的

这份文档面向下一次开发迭代，不是答辩稿，也不是产品宣传稿。

目标只有三件事：

1. 说明当前项目已经具备哪些可保留能力。
2. 明确前端和后端各自的结构性不足。
3. 给出下一次迭代最值得投入的方向、优先级和创新点。

评估范围仅包含：

- `frontend`
- `src/main/java`

---

## 2. 当前项目能力概览

### 2.1 前端现状

当前 `frontend` 已经不是单纯的接口联调页面，而是一套完成度较高的内容前台，具备这些能力：

- 首页：Hero、文章列表、热门侧栏、主题切换、转场动效
- 发现页：关键词搜索、分类筛选、排序切换
- 文章详情页：正文渲染、目录、评论区、收藏
- 用户体系页面：登录、注册、个人中心
- 创作页面：写文章、草稿、发布、预览
- 交互层：主题切换、动效偏好、页面转场、粒子背景、响应式布局

技术栈已经比较清晰：

- Next.js 16
- React 19
- SWR
- Zustand
- Tailwind CSS 4
- Framer Motion

### 2.2 后端现状

当前后端已经覆盖博客主流程：

- 认证：注册、用户登录、管理员登录
- 用户：资料读取与更新、我的文章、我的评论、我的收藏
- 文章：前台列表、前台详情、后台列表、后台详情、创建、更新、删除、置顶
- 评论：发表评论、一级评论、评论树、删除、回复
- 分类：分类列表
- 收藏：收藏、取消收藏

技术栈基础明确：

- Spring Boot 3
- MyBatis-Plus
- MySQL
- Redis 依赖已接入但业务上未充分使用

### 2.3 当前值得保留的亮点

当前系统最有价值的不是“功能很多”，而是已经具备继续打磨成作品的基础：

- 前端视觉完成度明显高于普通课程项目，具备独立展示价值。
- 文章、评论、收藏、分类、标签、草稿/发布、置顶等核心概念已经齐全。
- 前后端已经打通主要链路，博客闭环基本成立。
- 分类固定化之后，项目已经具备向“轻内容社区”演进的基础。

## 3. 前端不足

### 3.1 用户态依赖后端缺陷做补偿

当前前端并没有拿到标准登录态，而是在补后端能力的缺口。

具体表现：

- `frontend/app/login/page.tsx` 中必须手动填写 `userId`
- `frontend/lib/api.ts` 中 `userApi.login()` 只接收后端返回的一段字符串
- `frontend/lib/store.ts` 中使用 `knownAccounts` 和本地持久化来维持“可用用户态”

这类实现能联调，但不是可上线方案。

结论：

- 当前前端“能登录”，不等于“有完整认证体系”
- 下一次迭代必须把“联调用户态”升级成“标准登录态”

### 3.2 数据聚合职责过度前置到前端

前端目前承担了不少本该由后端 VO 或聚合接口承担的工作。

具体表现：

- `frontend/app/explore/page.tsx` 中分类来源于文章列表二次去重，而不是独立分类接口返回完整聚合数据
- `frontend/components/home/sidebar.tsx` 中热门文章、热门标签、活跃作者全部依赖前端对文章列表再次计算
- `frontend/components/profile/profile-tabs.tsx` 中“我的文章 / 我的收藏”先拿 ID，再逐个补详情

这会带来几个问题：

- 请求次数偏多
- 页面逻辑变重
- 接口契约脆弱
- 数据统计口径分散，后期难统一

### 3.3 页面能力仍偏联调式，不是完整产品形态

当前一些关键页面仍然处在“可以演示，但不适合长期用”的状态。

典型问题：

- `frontend/components/profile/profile-header.tsx` 中头像仍是 URL 文本输入，不是上传流程
- `frontend/app/write/page.tsx` 中标签仍通过逗号分隔 ID 文本输入，不是可视化标签选择
- 评论区 `frontend/components/article/comment-section.tsx` 仍只展示 `userId`，没有昵称、头像、角色等展示信息
- 文章详情页的评论更接近“底层数据回显”，而不是社区化互动体验

### 3.4 前端契约和校验仍然偏弱

虽然页面完成度高，但工程层还不够稳。

具体问题：

- 缺少表单 schema 级校验，主要依赖组件内手写校验
- 缺少组件测试、流程测试、页面级错误恢复
- 很多页面逻辑建立在“我知道后端目前是这样返回”的假设上，而不是稳定接口契约
- 图片和媒体只处理了 URL 展示，没有形成统一资产策略

### 3.5 前端最关键的短板总结

前端目前最大的不足不是界面，而是：

- 认证体系是假登录态
- 数据聚合过度前置
- 社区信息展示不够完整
- 媒体与上传能力缺失

这意味着它目前更像“高完成度联调前台”，而不是“可长期维护的产品前台”。

---

## 4. 后端不足

### 4.1 认证体系明显不完整

这是当前最明显的系统级短板。

现状：

- `src/main/java/com/cug/miniblog/personalCenter/controller/AuthController.java`
  - `/auth/login` 返回 `String`
  - `/auth/admin/login` 返回 `String`
- `src/main/java/com/cug/miniblog/personalCenter/service/impl/AuthServiceImpl.java`
  - 登录成功返回用户名
  - 没有 token
  - 没有 session
  - 没有当前用户接口
  - 密码仍是明文比较

这直接导致：

- 前端必须手动绑定 `userId`
- 没法做标准权限控制
- 没法做真正的管理员边界
- 不适合公网环境

### 4.2 接口返回契约不统一

当前接口风格明显混杂。

具体表现：

- 文章接口主要返回 `Result`
- 认证接口直接返回 `User` 或 `String`
- 用户资料更新、收藏接口直接返回 `boolean`
- 评论列表直接返回 `List<Comment>` 或 `HashMap<Long, List<Comment>>`

这导致的问题是：

- 前端需要写大量兼容代码
- 错误处理方式不统一
- 新功能难以快速接入

### 4.3 用户侧接口颗粒度偏底层

当前用户侧很多接口只够“拼页面”，不够“直接渲染页面”。

具体表现：

- `/users/articles` 返回 `List<Long>`
- `/users/comments` 返回 `List<Long>`
- `/users/favorites` 返回 `List<Long>`
- 评论接口直接返回 `Comment` 实体，不带用户名、头像、回复目标、展示层字段

这意味着：

- 前端必须补拉详情
- 评论展示无法形成完整用户关系链
- 用户中心始终偏“技术联调页”

### 4.4 业务语义存在不一致

后端当前有几处会直接影响后续迭代的语义问题。

典型例子：

- `UsersServiceImpl.updateProfile()` 中用 `avatar.matches(BIO_PATTERN)` 校验个人简介规则，字段校验明显写错
- `UsersServiceImpl.getStatusValue()` 中把 `pending/published/failed` 映射成 1/2/3，但文章服务里主流程只认可 0/1 状态
- `CommentsServiceImpl.replyComment()` 中查询回复目标时使用 `commentId + userId` 联合匹配，逻辑上会把“只能回复自己的评论”这种错误限制带进来

这些问题说明：

- 现在的后端已经不是“完全没做”，而是进入了“逻辑开始相互影响，需要收敛规则”的阶段

### 4.5 工程健康度不稳定

当前仓库的一个核心问题是：**业务代码存在，但工程稳定性不足**。

已确认现象：

- `mvn compile` 不能干净通过
- 编译报错集中在 DTO / VO / 实体 / Lombok / 方法引用这几层
- 说明当前后端代码不是稳定可持续开发状态

这类问题会直接影响下一次迭代，因为：

- 新功能越多，编译链越脆
- 团队协作时会持续被环境/注解处理问题卡住

### 4.6 后端能力缺口

当前后端还没有形成“完整平台能力”，而是停留在“核心博客 CRUD”。

缺失能力包括：

- 文件上传
- 媒体资源管理
- 内容审核
- 通知/消息
- 推荐
- 搜索增强
- 标签/分类后台治理
- 创作者数据统计

这些恰好就是下一次迭代最值得补的位置。

## 5. 下一次迭代建议

下一次迭代不建议平均发力，而应分层推进。

---

### 5.1 P0：先补工程底座

这一层不做，后续功能都会继续建立在脆弱前提上。

#### 方向 1：统一认证与用户态

- 目标：把当前“联调登录”改成标准登录态
- 价值：解决前端复杂补偿逻辑，建立后续权限基础
- 成本：中
- 依赖：后端登录接口改造

建议内容：

- 登录返回标准响应对象
- 包含 token / userId / role / username
- 增加鉴权拦截
- 增加“当前用户”接口

#### 方向 2：统一接口返回与错误体系

- 目标：后端所有面向前端的接口统一 `Result`
- 价值：前端可大幅简化适配逻辑
- 成本：中
- 依赖：controller 层统一改造

建议内容：

- 统一成功/失败响应
- 统一错误码
- 统一 message / data / total 语义
- 增加全局异常处理

#### 方向 3：先恢复后端编译健康

- 目标：`mvn compile` 能干净通过
- 价值：保证后续迭代可持续
- 成本：中
- 依赖：DTO / VO / 实体 / Lombok 问题清理

建议内容：

- 清理 DTO/VO getter/setter 与注解处理问题
- 修正编译时方法引用错误
- 补最基础的单元测试或启动验证

---

### 5.2 P1：补“可上线”的基础产品能力

这一层解决的是“现在像一个项目，但还不像一个产品”的问题。

#### 方向 4：补媒体上传链路

- 目标：支持头像上传、文章封面上传
- 价值：显著改善用户资料页和创作页的真实度
- 成本：中
- 依赖：新增上传接口、静态资源或对象存储方案

建议内容：

- 头像上传
- 封面上传
- URL 文本输入改为上传流程
- 默认图与失败回退机制

#### 方向 5：重构评论为面向 UI 的展示模型

- 目标：让评论区从“数据回显”升级为“社区互动”
- 价值：直接提升文章详情页质量
- 成本：中
- 依赖：评论 VO、用户信息聚合

建议内容：

- 评论返回昵称、头像、角色、回复对象信息
- 修正回复逻辑和删除权限逻辑
- 明确一级评论与回复链展示结构

#### 方向 6：重构个人中心接口

- 目标：个人中心不再依赖“先拿 ID 再补详情”
- 价值：降低前端复杂度，提高接口稳定性
- 成本：中
- 依赖：新增用户中心 VO 接口

建议内容：

- `/users/articles` 直接返回文章摘要卡片数据
- `/users/favorites` 直接返回收藏文章卡片数据
- `/users/comments` 返回评论摘要、所属文章、时间等展示字段

#### 方向 7：增强后台内容治理

- 目标：后台从文章 CRUD 升级为内容治理入口
- 价值：让系统更像平台，不只是编辑器
- 成本：中
- 依赖：分类、标签、文章状态管理

建议内容：

- 分类管理
- 标签管理
- 草稿/发布流转规则明确化
- 文章状态语义统一

---

### 5.3 P2：做出版本亮点

这一层解决的是“下一版有什么记忆点”。

#### 方向 8：从课程博客升级为轻社区内容平台

- 目标：基于固定分类建立内容社区感
- 价值：最符合当前项目定位，也最容易讲清楚
- 成本：中
- 依赖：分类体系、作者信息、评论体验、推荐基础

建议内容：

- 按分类做兴趣分区
- 首页/发现页突出分区入口
- 文章详情增强作者信息与相关推荐

#### 方向 9：个性化推荐与创作者数据

- 目标：让内容流动起来，不再只是静态列表
- 价值：产品感最强
- 成本：中到高
- 依赖：阅读记录、收藏行为、文章统计

建议内容：

- 热门文章
- 同类文章推荐
- 创作者中心中的阅读趋势、收藏数、互动数

#### 方向 10：AI 创作辅助与内容治理

- 目标：做出具有辨识度的智能亮点
- 价值：展示价值高，也符合内容产品方向
- 成本：中
- 依赖：统一后端接口、上传/内容链路稳定

建议内容：

- 标题生成
- 摘要生成
- 标签推荐
- 评论/文章内容审核

## 6. 创新点建议写法

下一版不要写成“我要加 AI、我要做推荐”这种泛词，而要写成版本主题。

建议用下面三条作为创新主线。

### 6.1 产品主线

**从课程博客升级为轻社区内容平台**

强调点：

- 固定分类下的兴趣内容组织
- 作者、评论、收藏、推荐之间形成内容互动链

### 6.2 前端主线

**从接口联调型前端升级为可独立演示的完整产品前台**

强调点：

- 登录态标准化
- 上传、资料、评论、创作流程完整化
- 页面体验继续保持优势

### 6.3 后端主线

**从 CRUD 后端升级为具备内容治理与创作辅助能力的平台后端**

强调点：

- 统一契约
- 鉴权
- 聚合 VO
- 媒体处理
- 推荐与 AI 能力接入

---

## 7. 建议的下一版版本主题

如果只做一个版本，我建议下一次迭代的总主题定为：

**“把当前项目从高完成度博客 Demo，升级成可稳定演示的轻内容社区平台。”**

这个主题的好处是：

- 不会离当前代码太远
- 你已经有足够基础支撑它
- 既能解释为什么补工程底座，也能解释为什么做推荐、上传和 AI

---

## 8. 推荐优先级结论

如果只选最值得做的 6 个方向，建议顺序如下：

1. 标准登录态与鉴权
2. 统一接口返回与异常处理
3. 修复后端编译与状态语义不一致问题
4. 头像/封面上传
5. 个人中心与评论 VO 化
6. 分类社区化 + 推荐 + AI 写作辅助

这个顺序的逻辑是：

- 先修底座
- 再补完整产品链路
- 最后做亮点

如果顺序反过来，新增的推荐和 AI 只会建立在脆弱接口之上，后续维护成本会更高。

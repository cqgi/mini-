# 逆序队第八次组会纪要

- 会议时间：2026 年 4 月 16 日
- 参会人员：陈侨、张战胜、董志鹏、张志彤

## 这次主要聊了什么

这次会开得很像一次典型的缺陷复盘会。前两天实际联调时，评论回复这块出了一个很具体的问题：自己回复自己能成功，但回复别人时会失败。张战胜把后端逻辑修过之后，大家这次主要就是把前后端参数重新对齐，确认回复接口到底要带哪些信息，前端要怎么传，页面点哪条评论就该回复哪条评论。

另一个实际问题是演示环境。为了后面方便同一局域网内访问前端和后端，大家也把本地开发服务的访问范围、跨域和网络地址这些问题拿出来过了一遍。这个问题平时看着不大，真到演示前如果打不开页面，会比一般小 bug 更让人头疼。

## 每个人这几天推进到哪

- 陈侨这边把前端评论回复参数跟着后端修正，同时也在处理局域网访问和本地启动配置。
- 张战胜修掉了“评论别人失败”的缺陷，把回复逻辑真正改成能指向被回复用户。
- 董志鹏这边继续看登录返回信息，尤其是用户 `id` 这类前端很依赖的字段，避免页面里到处猜。
- 张志彤帮着从个人中心和收藏页面的角度看数据回填，确认前端页面在多设备访问时有没有明显问题。

## 联调时暴露了什么问题

- 评论回复这件事之前参数语义不够清楚，导致前端以为只传评论 ID 就行，结果真正还需要被回复用户的信息。
- 同一局域网访问时，前端 dev 资源和后端跨域限制如果没放开，很容易出现“本机能开，别的设备打不开”的情况。
- 登录接口如果不把 `id` 等关键字段返回清楚，前端很多操作只能靠猜，后面问题会越来越多。

## 其他人给前端的建议

- 张战胜明确告诉陈侨，评论回复这块前端一定要把“目标评论 ID + 被回复用户 ID”都带上，不然后端没法准确落回复关系。
- 董志鹏建议登录返回结构尽量稳定，至少让前端能直接拿到 `userId` 和角色信息，不要让页面再自己拼。
- 张志彤建议多用一两台设备试试页面，特别是个人中心和收藏列表，局域网访问一打开，很多布局问题会暴露得更快。

## 下次组会前要完成什么

- 把评论回复这条链路彻底测顺，别再留“某种情况能回、某种情况不能回”的问题。
- 把局域网访问、跨域、本地环境配置这些演示相关问题收掉。
- 开始整理后台管理能力和剩余未接完的前端部分，给最后收尾留时间。

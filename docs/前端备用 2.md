# Mini Blog 前端开发文档

## 1. 文档定位

这份文档用于指导前端开发，不是后端接口罗列版，也不是产品需求文档。

它的目标是回答下面几个问题：

- 前端到底要做哪些页面
- 每个页面依赖哪些接口
- 当前后端有哪些限制，前端该怎么绕开
- 现在的前端目录应该怎么继续扩展
- 联调时先做什么、后做什么

适用场景：

- 新同学接手前端
- 对照当前后端能力继续完善页面
- 做答辩演示前梳理页面与接口关系

## 2. 项目目标

本项目是一个 Mini Blog，当前前端需要覆盖三类场景：

- 门户端
  - 首页文章列表
  - 文章详情
  - 评论区
- 用户端
  - 注册 / 登录
  - 个人资料
  - 我的文章 / 我的评论 / 我的收藏
- 管理端
  - 管理员登录
  - 文章列表管理
  - 新建文章
  - 编辑文章

## 6. 页面与接口对照关系

这一节是前端开发最需要频繁看的部分。

### 6.1 首页 `/`

功能：

- 展示文章列表
- 标题关键词筛选
- 分类 ID 筛选
- 置顶状态筛选
- 分页

前端文件：

- [HomeView.vue](/Users/mamba/dev/java_project/mini-/frontend/src/views/HomeView.vue)
- [articles.js](/Users/mamba/dev/java_project/mini-/frontend/src/api/articles.js)

返回处理：

- 使用 `Result.data` 渲染列表
- 使用 `Result.total` 渲染分页

### 6.2 文章详情 `/posts/:id`

功能：

- 展示文章标题、正文、标签、作者信息
- 拉取一级评论
- 拉取评论树
- 发表评论
- 回复评论
- 收藏 / 取消收藏

接口：

- `GET /articles/{articleId}`
- `GET /blog-comments/blog/{articleId}/topCommentList`
- `GET /blog-comments/blog/{articleId}/commentTreeList`
- `POST /blog-comments/blog/post`
- `POST /blog-comments/blog/{commentId}/{userId}/reply`
- `DELETE /blog-comments/blog/{commentId}/{userId}/delete`
- `POST /users/favorites/{articleId}`
- `DELETE /users/favorites/{articleId}`

前端文件：

- [ArticleDetailView.vue](/Users/mamba/dev/java_project/mini-/frontend/src/views/ArticleDetailView.vue)
- [comments.js](/Users/mamba/dev/java_project/mini-/frontend/src/api/comments.js)
- [users.js](/Users/mamba/dev/java_project/mini-/frontend/src/api/users.js)

前端注意：

- 收藏、评论都依赖 `userId`
- 当前后端没有 token，所以前端必须从本地会话里拿 `userId`
- 评论回复接口是 `text/plain`，不是 JSON

## 8. 当前后端限制与前端处理策略

### 8.1 无 token / session / 鉴权拦截

后端现状：

- 登录只返回字符串
- 没有 JWT
- 没有当前用户接口

前端处理策略：

- 使用登录页手动填写 `userId`
- 本地 store 保存联调会话
- 所有依赖用户身份的接口，都从 store 中读取 `userId`

### 8.2 返回结构不统一

后端现状：

- 一部分直接返回对象或布尔值
- 一部分返回 `Result`

前端处理策略：

- 在 `http.js` 做统一适配
- 页面层尽量不直接判断后端差异

### 8.3 “我的文章 / 我的评论 / 我的收藏”只返回 ID

后端现状：

- 不能直接渲染完整卡片

前端处理策略：

- 我的文章：补拉详情
- 我的收藏：补拉详情
- 我的评论：先展示评论 ID，不伪造正文信息

### 8.4 分类、标签缺接口

后端现状：

- 没有分类列表接口
- 没有标签列表接口

前端处理策略：

- 后台编辑页先手动输入分类 ID 和标签 ID

### 8.5 评论结构偏底层

后端现状：

- 一级评论和评论树分两个接口
- 返回的是 `Comment` 实体，不带用户昵称和头像

前端处理策略：

- 先把评论树结构接通
- 展示层面先用 `userId` 占位

### 8.6 收藏使用逻辑删除

后端现状：

- 取消收藏不是物理删除，而是 `is_deleted = 1`
- 再次收藏时应恢复旧记录

前端影响：

- 不需要额外处理唯一索引问题
- 正常调用收藏接口即可

## 9. 当前前端目录职责

职责：

## 10. 建议开发顺序

如果继续扩展前端，建议按下面顺序推进。

### 第一阶段：保证主流程稳定

- 注册
- 登录
- 首页文章列表
- 文章详情
- 评论发布
- 收藏 / 取消收藏

### 第二阶段：完善用户侧

- 个人资料页优化
- 我的文章页补更多展示字段
- 我的收藏页增加空状态和批量操作
- 我的评论页等后端补 VO 后升级

### 第三阶段：完善后台侧

- 后台文章列表 UI 优化
- 编辑器体验优化
- 封面预览
- 表单校验强化

### 第四阶段：联调与收尾

- 接口异常提示统一
- loading / empty / error 三态统一
- 路由守卫
- 页面权限提示

## 11. 页面开发验收标准

前端每个页面至少满足下面几点：

### 11.1 页面能打开

- 路由存在
- 不白屏
- 不报控制台错误

### 11.2 状态完整

- loading
- empty
- error
- success

### 11.3 接口对接清晰

- 页面不直接写接口地址
- 接口失败有明确提示
- 接口成功后 UI 有变化

### 11.4 与后端限制保持一致

- 不假设后端有 token
- 不假设后端返回完整对象
- 不假设评论里有用户昵称和头像

## 12. 联调建议

推荐联调顺序：

1. 注册一个用户
2. 登录时填入真实 `userId`
3. 到个人资料页确认 `/users/profile` 可用
4. 到首页拉文章列表
5. 进入文章详情页测试评论与收藏
6. 打开我的收藏验证收藏列表
7. 打开后台页面测试创建 / 更新 / 删除 / 置顶

推荐重点检查：

- 登录后刷新页面，会话是否还在
- 收藏后刷新详情页，状态是否正确
- 评论树是否能正确挂载
- 后台页面刷新是否不会再误打到后端 JSON

## 13. 后续建议

如果后端后续继续完善，前端最值得升级的地方是：

- 增加真正的鉴权体系
- 增加分类 / 标签接口
- 增加评论展示 VO
- 增加“我的评论 / 我的文章 / 我的收藏”的详情型接口
- 增加统一错误码和统一响应结构
